# Stripe Radar Integration Guide

This guide covers integrating [Stripe Radar](https://docs.stripe.com/radar)
fraud signals using the Spreedly Web SDK. Radar is Stripe's fraud-detection system; 
it scores each transaction using signals from Stripe's
network plus rules you configure in the Stripe Dashboard.

## Overview

When a card is collected through Spreedly (hosted fields / express checkout) and
tokenized into Spreedly's vault, Stripe never sees the shopper's browser, so it
loses the client-side signals Radar relies on. A **Radar Session** closes that
gap: Stripe.js collects the browser signals client-side and returns a session id
(`rse_…`) that you forward with the payment so Stripe can re-associate those
signals with the server-side charge.

The SDK exposes a small, API-only surface for Radar:

1. The merchant loads Stripe.js on the page and calls `sdk.stripeRadar(publishableKey)`.
2. The SDK wraps `Stripe.createRadarSession()` and resolves with the session id
   (or `null` if a session could not be created).
3. The merchant forwards the session id to their backend alongside the payment
   method token.
4. The merchant's backend runs a purchase/authorization through the **Stripe
   Payment Intents gateway**, passing the id as a gateway-specific field. Stripe
   then evaluates the charge with Radar (visible in the Stripe Dashboard).

```
┌────────────┐  stripeRadar(key)   ┌─────────────────┐  createRadarSession  ┌────────────┐
│  Merchant  │ ──────────────────▶ │  Spreedly SDK   │ ───────────────────▶ │ Stripe.js  │
│  Frontend  │ ◀────────────────── │                 │ ◀─────────────────── │ (browser)  │
└─────┬──────┘   rse_… (or null)   └─────────────────┘   { radarSession }   └────────────┘
      │ radar_session_id + token
      ▼
┌────────────┐  POST /v1/gateways/{stripe_pi_gw}/purchase   ┌──────────────┐    ┌───────────┐
│ Merchant   │  gateway_specific_fields.stripe_payment_      │ Spreedly Core│ ─▶ │  Stripe   │
│ Backend    │ ───────────────────────────────────────────▶ │              │    │  + Radar  │
└────────────┘            intents.radar_session_id           └──────────────┘    └───────────┘
```

## Prerequisites

- Spreedly account with a [Stripe Payment Intents gateway](https://developer.spreedly.com/docs/stripe-payment-intents-gateway-guide) configured
- A Stripe **publishable key** (`pk_…`) belonging to the **same Stripe account**
  as that gateway — otherwise Stripe cannot correlate the session with the charge
- Radar enabled on your Stripe account
- Spreedly environment key and API credentials
- Auth params (nonce, timestamp, signature, certificate_token) generated by
  your backend

---

## Step 1: Load Stripe.js and the SDK

Stripe.js must be present on the page — the SDK does not inject it. Add it
alongside the SDK script:

```html
<script src="https://js.stripe.com/v3/"></script>
<script src="https://core.spreedly.com/checkout/sdk/{version}/index.js"></script>
```

```javascript
const sdk = new SpreedlyHostedFields({
  environment_key: 'your_environment_key',
  certificate_token: 'your_certificate_token',
  nonce: 'generated_nonce',
  timestamp: 'generated_timestamp',
  signature: 'generated_signature',
});
```

> Both `SpreedlyHostedFields` and `SpreedlyExpressCheckout` expose `stripeRadar`
> — pick whichever class your app already loads. No hosted fields or
> express-checkout iframe is mounted just to use Radar; the method lives on the
> shared base class.

## Step 2: Create the Radar session

Call `stripeRadar` and await the session id. The natural place is right before
you submit the payment, so the signals reflect the moment of checkout:

```javascript
const radarSessionId = await sdk.stripeRadar('pk_test_...');

if (radarSessionId) {
  // forward radarSessionId to your backend with the payment method token
} else {
  // session could not be created (e.g. Stripe.js missing) — see console/telemetry.
  // You can still charge; the transaction simply won't carry Radar browser signals.
}
```

For [Stripe Connect](https://docs.stripe.com/connect), pass the connected
account id so the session is created on the right account:

```javascript
const radarSessionId = await sdk.stripeRadar('pk_test_...', {
  stripeAccount: 'acct_...',
});
```

`stripeRadar` never throws — it resolves with `null` and logs the reason if
Stripe.js is absent, the key is missing/invalid, or session creation fails. The
session id is returned to you only; it is **not** attached to tokenization
automatically.

## Step 3: Run the purchase from your backend

After you have the session id, send it to your backend and run a
purchase/authorization through the Stripe Payment Intents gateway, forwarding
the id as a gateway-specific field:

```http
POST https://core.spreedly.com/v1/gateways/{stripe_payment_intents_gateway_token}/purchase.json
Authorization: Basic base64(environment_key:access_secret)
Content-Type: application/json

{
  "transaction": {
    "payment_method_token": "...",
    "amount": 2500,
    "currency_code": "USD",
    "gateway_specific_fields": {
      "stripe_payment_intents": {
        "radar_session_id": "rse_..."
      }
    }
  }
}
```

> **Important:** Run the purchase from your backend, not from the browser.
> Your access secret must never be exposed client-side.

The result of Radar's evaluation (risk score, rule outcomes) appears on the
charge in the **Stripe Dashboard** — it is not returned in the API response.

## API reference

### `stripeRadar(publishableKey: string, options?: StripeRadarOptions): Promise<string | null>`

Creates a Stripe Radar session and resolves with its id. Resolves with `null`
(and logs the reason) instead of throwing when a session cannot be created.

#### Parameters

| Parameter        | Type                 | Required | Notes                                                          |
| ---------------- | -------------------- | -------- | -------------------------------------------------------------- |
| `publishableKey` | `string`             | yes      | Stripe publishable key (`pk_…`) from the same account as your Stripe gateway |
| `options`        | `StripeRadarOptions` | no       | See below                                                      |

#### `StripeRadarOptions`

| Field           | Type     | Required | Notes                                                  |
| --------------- | -------- | -------- | ------------------------------------------------------ |
| `stripeAccount` | `string` | no       | Connected account id (`acct_…`) for Stripe Connect     |

#### Returns

`Promise<string | null>` — the Radar session id (`rse_…`) on success, or `null`
on failure. Failure cases (each logged, and emitted as a
`stripe_radar.session_completed` telemetry event with a `failure_reason`):

| `failure_reason`          | Cause                                                        |
| ------------------------- | ----------------------------------------------------------- |
| `stripe_not_loaded`       | `window.Stripe` is undefined (Stripe.js not on the page)    |
| `missing_publishable_key` | No publishable key was passed                               |
| `stripe_init_failed`      | `window.Stripe(...)` threw or returned falsy                |
| `no_session_id`           | Stripe returned no `radarSession.id` (see `radar.error`)    |
| `create_session_threw`    | `createRadarSession()` rejected                             |

### Gateway-specific fields (backend)

Passed under `transaction.gateway_specific_fields.stripe_payment_intents`:

| Field              | Type     | Notes                                                           |
| ------------------ | -------- | -------------------------------------------------------------- |
| `radar_session_id` | `string` | The `rse_…` id from `stripeRadar()`                            |
| `skip_radar_rules` | `boolean`| Optional — selectively bypass Radar rules on a transaction     |

See the [Stripe Payment Intents gateway guide](https://developer.spreedly.com/docs/stripe-payment-intents-gateway-guide)
for the full field list.

## Migrating from the legacy iframe

The legacy iframe exposed `Spreedly.stripeRadar(publishableKey, callback, options)`.
The new SDK keeps the same inputs and the same "return the id, you forward it"
model — the only change is **callback → Promise**:

```javascript
// Legacy
Spreedly.stripeRadar('pk_test_...', (id) => { /* forward id */ }, { stripeAccount });

// Web SDK
const id = await sdk.stripeRadar('pk_test_...', { stripeAccount });
```

---

## Security notes

- The publishable key (`pk_…`) is safe to expose client-side; never use a secret
  key (`sk_…`) in the browser.
- Run the purchase — and the gateway-specific field — from your backend only.
- The merchant page must load Stripe.js (`https://js.stripe.com/v3/`); ensure
  your CSP allows `script-src https://js.stripe.com` and the frames Stripe.js
  creates (`frame-src https://js.stripe.com`).

## Sample app

A working end-to-end demo lives at `web-sdk-sample-app/src/static/stripe-radar/`.
It tokenizes a card with hosted fields, creates a Radar session via
`sdk.stripeRadar()`, and charges through the Stripe Payment Intents gateway via
`POST /api/v1/stripe-radar-purchase`, which forwards `radar_session_id` as the
gateway-specific field above. The result panel shows the session id and whether
it was forwarded to Stripe.
